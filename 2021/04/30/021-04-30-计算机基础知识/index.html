R1.Make Fewer HTTP Requests

减少HTTP请求

  • 使用图片地图–比较少用

    就是将超链接关联到图片上,图片的不同位置可以相应不同的事件(比如导航栏就是一张图片,根据点击位置的不同做出不同的请求)

    缺点:图片必须是连续的

    • 服务器图片地图

      用户点击的时候会将点击坐标作为参数传递给服务器做判断

    • 客户端图片地图

      通过映射HTML的Map标签来实现

      tag_area demo

  • 图片精灵(CSS Sprites)

    任何支持背景图片的HTML元素都支持.

    通过使用CSS中的background-position 属性,将HTML元素放到背景图片中期望位置上.

  • 使用内联图片(也就是Base64进行编码)

    格式: data:[<mediatype>[;base64],<data>]

    图片数据是有大小限制的,并且使用base64嵌入到HTML页面中会增大整体的下载量,并且无法利用到浏览器的缓存机制

    但是将Base64放在CSS规则中写入外部样式表中,就可以利用浏览器的缓存机制

  • 合并脚本与样式

R2.Use a (CDN)Content Delivery Network

使用CDN分发内容

如果应用程序Web服务器离用户更近,则HTTP响应时间将缩短.另一方面,如果组件Web服务器离用户更近,则组件分发的响应速度更快.将应用程序Web服务器分散开成本比较大(集群).将组件Web服务器分散开,就可以达到响应时间大幅减少的目的,且更加容易实现.

R3.Add an Expires Header

添加Expires头 (利用浏览器缓存技术)

Expires:Mon,15 Apr 2024 20:00:00 GMT意味着在这个日期前,组件都是有效的,不用从服务器上重新获取组件.
缺点:Expires使用一个特定的时间,要求服务器与客户端的时钟严格同步,另外过期日期需要经常检查,如果未来这一天到来,需要在服务器上配置另外一个新的日期

  • 可替代方法:Max-Age HTTP1.1引入

    Cache-Control:max-age=315360000可以以一个相对的时间来设置更新窗口期.组件从请求开始过去的秒数在这个范围外,浏览器就会重新发起请求组件;而如果再这个范围内,则直接使用缓存

    Notice: 如果Max-Age与expires同时出现的话,Max-Age会重写expires.

    考虑兼容性的话,会同时制定Max-Age与Expires头

  • 如果不想要缓存的话,确保用户获取的组件是最新版本的.可以在文件名后面加上一个时间戳(或者随机值)

  • 节约的时间都去哪里了

    有长久的Expires头会缓存组件起来,在后续的请求中直接从硬盘读取它,避免了一个HTTP的请求.

    Expires过期的haul,虽然浏览器还是有缓存的,但是在后续的请求中,浏览器会向原始服务器发送一个条件GET请求,判断是否可以继续使用缓存组件

R4.Gzip Components

压缩组件–开启Web服务器压缩

  • 可以减少HTTP响应的大小

  • Gzip是最理想的压缩方法

  • Web客户端通过HTTP请求中的Accept-Encoding 来表示对压缩的支持

    请求头: Accept-Encoding:gzip,deflate

  • 响应服务器看到这个请求头,使用客户端列举的压缩方法之一来压缩响应

  • 压缩是有成本的,服务器需要花费额外的CPU资源来完成压缩,而客户端需要对压缩文件进行解压缩

  • 代理缓存所遇到的问题

    • 如果到达代理的第一个请求来自于一个不支持gzip的浏览器,后面来的请求都会使用其缓存中的内容(未压缩)的内容进行响应,进而失去的压缩的机会.
    • 如果到达代理的第一个请求支持gzip压缩的浏览器,后面的请求都会是压缩的.这样会导致不支持gzip压缩的浏览器显示不正常

R5.Put Stylesheets at the Top

将样式表放在顶部

  • 将样式放在底部的话,会出现白屏现象 Demo

    原因:浏览器已经下载得到所需要的组件,但是在样式表加载完成之前,浏览器构建呈现树是一种浪费.所以浏览器选择显示白屏(延迟呈现)

    如果浏览器没有选择白屏的话,就会遇到无样式内容的闪烁问题(FOUC,Flash of Unstyled Content) .

    规范中也是提倡将CSS放在顶部,这样无关页面是如何加载的,页面都是逐步呈现的

  • 将样式表包含在文档中有两种方式,使用Link标签和使用@import规,但是推荐使用LINK标签

  • 使用@import规则可能导致白屏现象,因为这个规则会导致下载时候的无序性.即便是放在文档的HEAD标签也是一样.

  • 样式闪烁并不总是发生,IE会在新窗口加载页面,重新加载和作为主页面的时候发生白屏;而在点击超连接的时候,或者输入URL时出现FOUC(无样式内容闪烁);而火狐浏览器总是发生FOUC.

  • 因此发生FOUC或者白屏的话,因浏览器的不同而不同

R6.Put Scripts at the Bottom

将脚本放在底部

  • HTTP1.1建议浏览器从每个主机名并行下载两个组件(建议归建议,现代浏览器普遍比这个值高)

  • 并行下载数量是可以进行调整的,但是需要通过用户自行设置

  • 前端工程师与其依赖用户来修改浏览器设置,不如简单地使用CNAME(DNS别名)来将组件放在多个主机名中

  • 增加并行下载数量并不是没有开销的

  • 下载脚本的时候浏览器并行下载功能将会被禁止

    因为脚本可能改变页面的内容

    保证脚本能够正确地被执行.如果模块之间有依赖关系的话,不按照执行顺序就可能会导致JavaScript错误

R7.Avoid CSS Expressions

避免使用CSS表达式

  • 只有IE浏览器支持表达式
  • 推荐使用事件处理器来代替CSS表达式
  • 通常来说,需要用到CSS表达式的地方都可以使用事件处理器来代替

R8.Make JavaScript and CSS External

使用外部的JavaScript与CSS

  • 单纯从首次加载的速度来看,使用内联会更块,因为少了部分的HTTP请求

  • 但是在组件复用的情况下,使用外部的JavaScript与CSS的缓存优势更大

  • 使用外部的JavaScript与CSS可以拥有更为长久的Expires头,更好的压缩内容

  • 使用内联或者外部从三个基准来考虑:页面访问量,空缓存与完整缓存的比重,组件复用

  • 如果从三个基准来考虑结果,得到缓存组件可以带来更大的受益的话,那么就要使用外部JavaScript与CSS

  • 下面两种办法让你即可以获得内联的优势,同时可以缓存外部文件

    1. 加载后下载

      • 加载后下载(使用onload事件,在document.onload之后一秒延迟执行,创建JavaScript/CSS DOM元素插入文件)
      • JavaScript与CSS会被加载到页面两次(先内联然后外部)
      • 必须处理双重定义:如脚本可以定义但是不能执行任何函数(至少不能让用户察觉到);使用CSS的相对单位指定两次可能会产生问题
      • 将组件放在一个不可见的IFrame中或许会更好

    2. 动态内联

      • 使用Cookie做指示器

        每个用户一开始都是没有cookie的,当用户第一次访问页面时候,服务器发现没有cookie,于是生成一个内联了组件的页面,并且服务器添加了JavaScript脚本来在页面后加载后下载(上面讲的)\外部样式.当下一次访问页面时,服务器看到了cookie,生成一个*只包含外部文件*的页面

R9.Reduce DNS Lookups

减少DNS查找

  • 通常浏览器查找一个给定的主机名需要用到20-120ms

  • 浏览器拥有自己的DNS缓存

  • 当浏览器找不到DNS请求时,会去寻找操作系统的DNS缓存

  • RFC建议TTL为一天,但是大部分网站发送给客户端的最大TTL在1min~60min之间

    设置那么短的原因是方便服务器发生故障时快速转移

  • 客户端收到的DNS记录的平均TTL值只有最大TTL值的一半,因为DNS解析器自身也用友与DNS相关的TTL.

  • 作者建议使用Keep-Alive和较少的域名来减少DNS查找

R10.Minify JavaScript

精简JavaScript

  • 精简代码(Minification):移除所有注释与不必要的空白字符

  • 混淆代码(Obfuscation):与精简代码一样,会移除注释与不必要的空白.同时改写代码,改写函数名称及变量到最精简

  • 节后gzip压缩之后,精简代码与混淆代码的区别会变小

  • 一般我在工作中,线上环境用混淆

  • 一般也会对CSS进行精简(不过带来的节省不如JavaScript明显)

    使用缩写
    移除不必要的字符串(eg:use 0 replace 0px)

R11.Avoid Redirects

避免重定向

  • 使用meta refresh标签页可以实现重定向,content来指定间隔时间后跳转
  • 使用JavaScript来执行重定向:document.location设置
  • 重定向会延迟整个HTML文档的传输,在用户和HTML文档之间插入重定向会延迟页面中所有东西

OSI与TCP/IP各层的结构与功能,都有哪些协议?

Application-layer(DNS , HTTP)

The application layer is responsible for process-to-process communication across the Internet. It is the topmost layer in the hierarchy, and the application itself is built on top of this. The application layer provides a communication interface and end-user services to the application for its communication with single processes.

任务是通过应用进程间的交互来完成特定网络应用。应用层协议定义的是应用进程（进程：主机中正在运行的程序）间的通信和交互的规则。对于不同的网络应用需要不同的应用层协议。在互联网中应用层协议很多，如域名系统DNS，支持万维网应用的 HTTP协议，支持电子邮件的 SMTP协议等等。我们把应用层交互的数据单元称为报文。

Transport layer(TCP, UDP)

The transport layer is responsible for connecting applications on the Internet. It demultiplexes data coming in from a single source and transmits it to the application it is intended for. The basic purpose of the transport layer in the context of the web is that it provides process-to-process communication; it allows two individual processes on either the same machine or separate machines to send messages to each other. To do so, it uses sockets, which are essentially just the gateway to a process. In other words, sockets are the means through which messages are received and sent out by a process.

主要任务就是负责向两台主机进程之间的通信提供通用的数据传输服务。应用进程利用该服务传送应用层报文。“通用的”是指并不针对某一个特定的网络应用，而是多种应用可以使用同一个运输层服务。由于一台主机可同时运行多个线程，因此运输层有复用和分用的功能。所谓复用就是指多个应用层进程可同时使用下面运输层的服务，分用和复用相反，是运输层把收到的信息分别交付上面应用层中的相应进程。

Network layer(IP)

The network layer is responsible for connecting any two machines on the Internet. It provides global connectivity and allows for end-systems to communicate with one another on a large scale, beyond what the data link layer has to offer.

在 计算机网络中进行通信的两个计算机之间可能会经过很多个数据链路，也可能还要经过很多通信子网。网络层的任务就是选择合适的网间路由和交换结点， 确保数据及时传送。 在发送数据时，网络层把运输层产生的报文段或用户数据报封装成分组和包进行传送。在 TCP/IP 体系结构中，由于网络层使用 IP 协议，因此分组也叫 IP 数据报 ，简称 数据报。

Internet Protocol, more commonly known as IP, is a network layer protocol that is responsible for assigning addresses to devices in order to give them unique identities that make them reachable and discoverable. Each device on the Internet has a unique IP address that other devices use to connect with it.

Link layer

The data link layer is responsible for transmitting data from any given machine to a device or machine that is exactly one link away.

链路层。两台主机之间的数据传输，总是在一段一段的链路上传送的，这就需要使用专门的链路层的协议。 在两个相邻节点之间传送数据时，数据链路层将网络层交下来的 IP 数据报组装成帧，在两个相邻节点间的链路上传送帧。每一帧包括数据和必要的控制信息（如同步信息，地址信息，差错控制等）。

TCP

The primarily used protocol for communication between a web application and a browser is referred to as the Transmission Control Protocol (TCP). TCP is a transport layer protocol that takes the responsibility of transmitting data and ensures reliable data transfer between clients and servers across the web. The way TCP does this is by adding additional information to data packets that allow for packet authentication and by allowing the exchange of acknowledgment messages between the client and server to confirm data transmissions.

TCP协议如何保证可靠传输

  1. 应用数据被分割成 TCP 认为最适合发送的数据块。
  2. TCP 给发送的每一个包进行编号，接收方对数据包进行排序，把有序数据传送给应用层。
  3. checksum校验和： TCP 将保持它首部和数据的检验和。这是一个端到端的检验和，目的是检测数据在传输过程中的任何变化。如果收到段的检验和有差错，TCP 将丢弃这个报文段和不确认收到此报文段。
  4. TCP 的接收端会丢弃重复的数据。
  5. flow control流量控制： TCP 连接的每一方都有固定大小的缓冲空间，TCP的接收端只允许发送端发送接收端缓冲区能接纳的数据。当接收方来不及处理发送方的数据，能提示发送方降低发送的速率，防止包丢失。TCP 使用的流量控制协议是可变大小的滑动窗口协议。 （TCP 利用滑动窗口实现流量控制）
  6. 拥塞控制： 当网络拥塞时，减少数据的发送。
  7. ARQ协议： 也是为了实现可靠传输的，它的基本原理就是每发完一个分组就停止发送，等待对方确认。在收到确认后再发下一个分组。
  8. 超时重传： 当 TCP 发出一个段后，它启动一个定时器，等待目的端确认收到这个报文段。如果不能及时收到一个确认，将重发这个报文段。

ARQ协议

自动重传请求（Automatic Repeat-reQuest，ARQ）是OSI模型中数据链路层和传输层的错误纠正协议之一。它通过使用确认和超时这两个机制，在不可靠服务的基础上实现可靠的信息传输。如果发送方在发送后一段时间之内没有收到确认帧，它通常会重新发送。ARQ包括停止等待ARQ协议和连续ARQ协议。

TCP如何实现流量控制？Sliding Window & flow control

TCP 利用滑动窗口实现流量控制。流量控制是为了控制发送方发送速率，保证接收方来得及接收。 接收方发送的确认报文中的窗口字段可以用来控制发送方窗口大小，从而影响发送方的发送速率。将窗口字段设置为 0，则发送方不能发送数据。

Conjestion control拥塞控制

拥塞控制主要由四个算法组成：慢启动（Slow Start）、拥塞避免（Congestion voidance）、快重传 （Fast Retransmit）、快恢复（Fast Recovery）
