AE Tips&Tricks

Communication

  1. Eliminating ambiguity
    • eg. at start, when you’re given the question, a lists of number, you have to do sth about it. Only integers or am I also dealing with decimal numbers. Positive/negative numbers. Can we have duplicate numbers? Can the array be sorted?
  2. Voicing your thought progress
    • you don’t want your interviewers to guess things
  3. Discussion trade-off
    • eg. In first solution, we transform this array into a linkedlis `t.
    • In other solution we just keep the array as a typical array.
    • The linkedlist approach will be good because it would allow us to add values in linear time at any position in the linkedlist ,whereas the array would allow us to add values only in O n squared time where n is the length of the input. But on the other hand the linked list approach has a lot of overhead because we have to build the linked list at the begining. Whereas the array we’re kind of giving it for free. Overall I think I’m gonna favor the linkedlist approach because despite the overhead, it’ll simplify things the further down we are in the problem.
  4. Proactively asking for approval or help form your interviewer
    • Can I use IMI for this variable . Is the array multible?
  5. Warning.
    • Do not over do these thing, keep a balance

Problem Solving Ability

  1. Asking clarifying questions to eliminating ambiguity
  2. Take some time to discover the problem, digest the question. Do not just jump to the question rush, take some time to explore the problem and identify the hidden problem
  3. Discussing trade-off. Consider alternatives, consider pros and cons, trade-off, iterative or recursion method, choose which data structure, algorithm is better
  4. Before coding, voice your interviewer what you gonna do, tackle the problem in an organized manner
  5. Proactively testing the problem. In the end. Now I gonna explain how I would test my solution. Here are the edge cases that I would test for.

Tips:

  1. Come up with a brute-force solution
  2. Think of a simpler version of the problem
  3. Think with simpler examples -> try noticing a pattern
  4. Use some visualization
  5. Test your solution on a few examples

How to best use AE?

  1. Group, easy-difficult
  2. Group by category
  3. Group randomly
  4. Try to come up the solution by you own

Coding Skills

  1. Descriptive variable naming
  2. Abstraction, make use of helper method, separate your code into these logical units
  3. Documentation
  4. Descriptive code, change the condition into English
  5. Idiomatic coding style
  6. Testing

Culture Fit

  1. Show you're passionate about software engineering or about product development
  2. Pick a unique aspect or product of the company and show interest in it
  3. Introduce yourself. I am someone who’s always been interested in entrepreneurship and product management , and you know building cool things. But throughtout college I had this misconception that coding was something you had to have been doing since you were a kid. So I didn’t code during college. Coming out of college I realized, hey this is something I really need. Otherwise I’m not gonna be able to achieve the things I wanna do from a career point of view. I enrolled in xxx, fell in love with software engineering. And now here I am.
  4. Describe your past projects and jobs well. Everything in your resume should be explain well. Try to sell yourself. What did you do at the past job/ college/ teach yourself computer science?
  5. 逆问题What’s been your biggest challenge on the job? What’s the day in the life of you at this company?

What To Do If You’re Stuck In A Coding Interview

Reiterate the entire thinking progress talk out louder:

Problem statement, some constraints, edge cases

Approach to solve the program

Data structure, reason use or not use

eg. The program give us an array of integers and it want us to do x, we are told array is sorted and now

General Tips

Avoid

  1. Do not be disrespectful, offensive, mean, arrogant, confrontational.
  2. Avoid being too vague in your answers. Share enough information, go into enough details about your past experiences and your past stories, explain how you handle, who you work with, how you reacted to certain events, and what you learned from these events

Want to do

  1. Strike the balance between highlighting very good qualities (you are amazing, you can write complex code, you can lead the project )and your ability work in a team, the ability to help other. Stories highlight both these two things.
  2. Well you know, I don’t think I actually have this king of experience, but I’s love to tell you how I would react or how I would behave in the circumstance.
  3. General communication skills. Be able to convey the stories of past experiences in a concise manner. More prepare, more confidence. Record yourself and just rehearse some of these answers to the questions multiple times.

AT week1

What is an interview about?

  1. Image you are a sole founder of the company and ready to hire the first employee. How would you design the interview?
    • Find out if the candidate will be an awesome future colleague
    • If we really want to work with you and we feel excited to have you on the team, you are definitely hired. Looking for both positive and negative signals.

What signals to look for in a coding interview?

  1. Image you are a hired great engineer. You really enjoy working with this person. What are the traits that make them awesome to work with
  2. Their code is easy to read and understand. When you need to read their code, you can understand what they try to achieve and how they achieve it with the code they wrote.
  3. They can help you turn a problem into clear logic. When you have a messy problem and don’t know where to start you consult them. They bring you clarify. You feel the logic is so obvious that you can just turn it into clean code
  4. You feel they understand you. Sometimes you have a problem that you can’t even clearly describe what you want. You talk to them. They ask a few questions to learn from you. And then they describe your problem in a much better way.
  5. You feel comfortable delegating work to them. They don’t keep you in the dark. They explain how they are going to work succinctly. They deliver what they promise. They update you when they need to change the plan
  6. They respond to your feedback in a positive way, When you point out where their work can be improved, they focus on how to achieve that improvement with you. They don’t become overly defensive.

Summary of the signals

  1. Communication
    • Succinct articulation
    • Talking feedback
  2. Problem solving
  3. Coding
    • Language proficiency
    • Code quality
  4. Debugging and testing

BFS v.s. DFS for Path Finding

  • BFS returns path with shortest number of edges.
  • Time efficiency is similar for both algorithms.
  • Space Efficiency is quite different.
    • DFS is worse for spindly graphs. (Call stack gets very deep.)
    • BFS is worse for bushy graphs. (Queue gets very large.)
  • Track the distTo and edgeTo arrays requires Θ(V) memory.

cs61b week13

Prefix Operations and Tries

Tries is an ordered tree data structure used to store keys which can be broken into “characters” and share prefixes with other keys.

  • Every node stores only one letter.
  • Nodes can be shared by multiple keys.
  • The last character of the key is marked.

To check if the trie contains a key, walk down the tree from the root along the correct nodes. If the character does not exist or the final node is not marked, the key does not exist.

For example, the following trie contains ["sam", "sad", "sap", "same", "a", "awls"].

  • contains("a"): true, the final node is marked
  • contains("sa"): false, the final node is not marked
  • contains("saq"): false, fell off tree

Implementation

DataIndexedCharMap

The DataIndexedCharMap class is efficient implementation for a map which takes in character keys.

The value R represents the number of possible characters (128 for ASCII).

When building a trie, the DataIndexedCharMap class provides a map to all of a nodes’ children. However, the DataIndexedCharMap object of a node with relatively few children will have mostly null links, which is wasting excess memories.

Since each link corresponds to a character if and only if that character exists, the character variable of each node can be removed. The value of the character is its position in the parent.

Child Tracking

The problem of the implementation above is that it wastes excess memories with lots of null spots that don’t contain any children. However, alternative ways to track children is available.

  • Hash-Table based Trie: Initialize the default value and resize the array only when necessary with the load factor.
  • BST based Trie: Create children pointers when necessary and store them in a BST. Runtime is not efficient.

DataIndexedCharMap

  • Space: 128 links per node
  • Runtime: Θ(1)

BST

  • Space: C links per node
  • Runtime: O(log R)

Hash Table

  • Space: C links per node
  • Runtime: O(R)

(C is the number of children. R is the size of the alphabet.)

Performance

If a Trie has N keys, the runtime for our Map/Set operations are as follows:

  • add: Θ(1)
  • contains: Θ(1)

The runtime is independent from the number of the keys, since the operations only traverse the length of one key in the worst case.
